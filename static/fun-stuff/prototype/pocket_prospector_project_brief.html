COMP 4XXX — Final Exam Game Jam

Pocket Prospector

An incremental mining game where every dig is a discovery. Scratch through rock layers, collect rare minerals, and build an underground empire one vein at a time.

Genre Incremental / Idle Platform Web (HTML5) Engine Vanilla JS + Canvas Team Solo

Concept Overview

Pocket Prospector transplants the scratch-ticket interaction model from Scritchy Scratchy into a mining fantasy, while adopting Cookie Clicker's transparent progression economy. The player scratches through procedurally generated rock tiles to uncover minerals, gems, and fossils. Each "claim" is a unique tile grid with its own geology, reward table, and visual theme — the equivalent of a themed scratch ticket.

Where Cookie Clicker asks "how big is my number?", Pocket Prospector asks "what will I find next?" The process-focused engagement loop (the scratch) is the star; the outcome-focused loop (currency accumulation and reinvestment) provides long-term structure and prestige motivation.

Core Loop

Scratch rock tiles → uncover minerals → sell for gold → upgrade tools → unlock deeper claims → prestige to reset with permanent bonuses → repeat with new geology.

Design Pillars Philosophy

Process over Outcome

Every scratch must feel good. Variable reward, progressive disclosure, and tactile audio feedback make the dig itself the primary source of satisfaction — not just the number it produces.

Layered Information

Surface play is visual and intuitive (scratch, see colors, collect). Optimization play reveals probability tables, hardness stats, and efficiency calculations — available on demand, never forced.

Catalog as Progression

New claim types unlock in themed catalogs (Shallow Soil, Deep Stone, Volcanic Core, Fossil Beds). Progression feels like discovery and collection, not a linear store scroll.

Modular Architecture

Every system — claims, minerals, upgrades, prestige — is a self-contained module with a clean interface. New content ships as data, not code changes.

Systems Architecture Technical

The codebase is organized as a modular event-driven system. Each module registers with a central EventBus and communicates through typed events. No module directly imports another — all coupling flows through events and a shared state store.

ClaimEngine

Generates tile grids from geology templates. Handles scratch input, hit detection, hardness calculation, and reveal animation triggers.

EconomyManager

Tracks currencies (gold, gems, fossils). Processes transactions, validates purchases, and emits balance-change events for UI binding.

UpgradeRegistry

Defines upgrade trees (tool power, scratch radius, luck multiplier). Each upgrade is a data object with cost curve, effect function, and max level.

CatalogSystem

Manages claim-type unlocks, catalog theming, and collection tracking. Gates content behind prestige tiers and discovery milestones.

PrestigeLoop

Handles soft resets. Calculates prestige currency from lifetime earnings, applies permanent multipliers, and unlocks new catalog tiers.

SaveManager

Serializes game state to localStorage with versioned schema. Handles migration between versions and export/import for backup.

// Module registration pattern
class ClaimEngine extends GameModule {
  init(bus, store) {
    bus.on('claim:scratch', (pos) => {
      const tile = this.revealTile(pos);
      bus.emit('mineral:found', tile.contents);
    });
  }
}

UX Principles Design

From Scritchy Scratchy: Atmosphere & Process

The play area is a miner's workbench — a skeuomorphic surface with a lantern, field journal, and specimen jars. Claims sit on the bench as physical rock slabs. The environment is decorative but functional: the field journal tracks collection progress, the specimen jars show recent finds, and the lantern brightness reflects current luck multiplier.

From Cookie Clicker: Transparency & Legibility

Currency, production rate, and upgrade costs are always visible in fixed sidebar panels — never hidden behind hover states for core metrics. The upgrade panel uses Cookie Clicker's linear-list clarity: name, cost, level, and effect are all visible at a glance. The player can always answer "what should I buy next?" without opening a submenu.

Synthesis: Best of Both

The central workspace is atmospheric and tactile (Scritchy Scratchy's philosophy). The peripheral UI is transparent and information-dense (Cookie Clicker's philosophy). The boundary between these zones is sharp and consistent: workbench = experience, sidebars = information. This spatial contract lets players shift between process-focused and outcome-focused play by moving their eyes, not navigating menus.

Development Timeline Schedule

Week 1–2
Core Engine & Scratch Mechanic
EventBus, state store, ClaimEngine with canvas-based scratch interaction. One claim type, one mineral table. Scratch feels satisfying before anything else is built.
Week 3
Economy & Upgrades
EconomyManager, UpgradeRegistry, sidebar UI. Three upgrade paths (power, radius, luck). Cost curves balanced against a single claim type. Cookie Clicker-style always-visible upgrade list.
Week 4
Catalogs & Prestige
CatalogSystem with 3 themed catalogs (8+ claim types total). PrestigeLoop with soft-reset economy. Collection tracking in the field journal. SaveManager with versioned schema.
Week 5
Polish, Balance & QA
Sound design, particle effects on rare finds, workbench decoration. Full economy balance pass. Playtest feedback integration. Performance optimization for mobile canvas.

Evaluation Criteria Grading

Modularity (25%)

Can a new claim type be added by writing a single data object? Can a new upgrade be registered without modifying existing modules? The architecture should pass the "new content = new data, not new code" test.

Design Philosophy Integration (25%)

Does the game demonstrate understanding of both Scritchy Scratchy's process-focused UX and Cookie Clicker's outcome-focused transparency? Are the lessons from the essay visibly applied?

Player Experience (25%)

Is the scratch satisfying? Is progression legible? Does the first minute hook and the first hour sustain? Balance, feel, and juice matter here.

Technical Quality (25%)

Clean code, consistent patterns, no global state leaks, performant rendering, save system resilience. The codebase should read as well as it plays.
