Hosted Service — synthograsizer.com is live
  • feat
    Sign in with Google and use the suite in the browser — 300 credits a month, free
    The suite now runs as a hosted service at synthograsizer.com, backed by Cloud Run in Montréal. Signing in with Google grants 300 credits per month (text costs ⚡1–5, images ⚡4–15), topped back up on your first visit each month — no rollover, no card, no waiting list. Anonymous visitors can still open the app and play with templates, variables, and knobs; only the AI calls require an account. Local installs are completely unaffected — run it yourself with your own API key and none of this applies.
  • feat
    Your data stays yours — one-click export and account deletion
    The account menu has Download my data (a JSON file of your account, credit history, and generation log) and Delete account, which removes everything immediately — no email round-trip, no waiting period. Prompt text is never stored: the generation log keeps only metadata like which model ran and how long it took, and it ages out automatically on a retention timer. Anything you explicitly save to My creations (see below) is the one exception — it's kept until you delete it, not on a timer, since that's the point of saving it.
  • feat
    Video and music studios are local-only for now
    Veo, Lyria, and Videorama stay off on the hosted tier — a single Veo clip can cost more than a month of everyone's text generation combined. Those tools are hidden rather than teasingly disabled, and the underlying endpoints refuse the call too. Run the suite locally with your own key and they all work as before. A paid tier that unlocks them is designed but not built.
  • fix
    Launch buttons open the real app, not the gated demo
    Every "Launch" call-to-action on the landing and about pages now goes straight to the full suite. The demo build had been the default for public visitors back when there was no way to sign in; the free tier replaces it. The gated demo is still reachable from the suite navigation if you want it.
  • fix
    Backend & Safety panel now talks about credits, not a mythical "instance operator"
    Settings → Backend & Safety carried notices written before the service existed — a generic operator managing config "via environment variables," which means nothing if you're signed in and paying in credits. The panel now branches on whether accounts are on: with accounts, it explains metering and points at the account menu; a plain hosted deployment without accounts keeps the operator wording. The backend note also stopped hardcoding "Google" — it names whichever tier is actually pinned, so a hosted local-model deployment no longer claims Google's usage policy applies to it.
  • fix
    Smart Video Options and Send to Scope no longer show up as dead ends on the hosted service
    Both buttons sat on every image result and led nowhere for free-tier users. Smart Video Options now shows only for admins, matching the admin-only video endpoint it opens; Send to Scope hides for everyone in service mode, admins included — it bridges to a renderer on your own machine, and the underlying endpoint 403s before a hosted request could ever reach one.
  • fix
    Fixed a database migration bug that would have broken every fresh install
    Laying groundwork for a future account-storage feature surfaced a latent bug in the service database's upgrade path: a brand-new database computed itself as version zero and replayed every migration step up to current, even though it had just been built at the current schema in one shot. Harmless while every step was an additive table creation, but the first column-add or backfill would have errored or double-applied on new installs. Fresh databases now stamp straight to the current version with nothing replayed; migrations only run against a database that already has its own older recorded version. Covered by a new test written to fail against the old logic first.
Saved Creations — the "My creations" gallery
  • feat
    Save any generated image, video, or track to your account — find it later under My creations
    A 💾 Save button now sits beside generated images and videos. Saved items live in Cloud Storage (same Montréal region as everything else), private to your account, and stay there until you delete them — no retention timer, since that's the point of saving something. The account menu's new My creations entry lists everything you've saved with size and date, opens any item on demand, and deletes individually. A quota meter (200 MB by default) keeps things bounded; deleting your account removes every saved item along with it, immediately.
  • feat
    Download button beside Save on generated images — no account needed
    A ⬇ Download button now sits next to Save on every generated image and puts the PNG straight in your downloads folder, named by date and time. Unlike Save it needs no account and works on local installs, because nothing is uploaded — the picture is already in your browser, which is how you're looking at it. One thing worth knowing, and it's why the button says so on hover: generated PNGs carry the prompt that made them in the file's metadata, so sharing the image shares the prompt. Strip the metadata first if that matters for a particular image.
  • feat
    My creations now shows thumbnails
    Saved images get a little preview in the gallery instead of a generic icon. The thumbnail is made in your browser when you save (a small ~256px copy), so opening the list stays a single quick request and previews only load as you scroll to them. Older images saved before this update keep showing an icon; anything you save from now on gets a thumbnail. Videos and tracks still show their icon.
  • feat
    Save a generated template to your library — and load it back in one click
    When you generate a template, there's now a 💾 Save to My creations button, and it comes pre-named: a quick one- or two-word theme name so you can spot it later (e.g. "Neon Cats"). Saved templates live in My creations alongside your images; each one gets a Load template button that drops it straight back into the app. Saving is always a deliberate click — nothing is auto-saved. One honest note: a template is prompt text, so unlike an image (where only the file is stored), saving a template does keep its prompt on our server — that's spelled out in the updated Terms §7, and it only happens when you click Save.
  • fix
    Agent Composer no longer dead-ends with "Launch failed" on the hosted service
    Live multi-agent sessions (the Agent Composer / Agent Studio chat rooms) run on a separate backend that only exists on local installs, so launching one on the hosted service failed with a cryptic chatroom_unavailable error after you'd built the whole session. It now says up front that live sessions are local-only, before you do the work — and the rest of the Composer, including generating agent profiles and templates, keeps working on the hosted service. Running locally? Nothing changes; sessions launch as before.
  • chore
    Terms bumped to v0.3 — saved creations are the first hosted feature that stores generated media server-side
    Everything else about the service is unchanged: prompts are still never stored, and nothing is saved unless you click Save. Since this is a real change in what the service does with your content, signing in again shows the terms interstitial once more so you can review and re-accept — not a bug, just consent catching up to the feature.

Gemini Interactions API Migration
  • feat
    All Gemini calls migrated to Google's Interactions API — stateless by default
    Text, chat, vision, multimodal template modes, Gemini image generation, and the entire ChatRoom server now use interactions.create (Google's GA interface, June 2026) instead of generateContent. Every request is sent with store=false — nothing is retained server-side at Google (the API's default would keep requests for 55 days). The deprecated google-generativeai / @google/generative-ai SDKs are gone; Veo, Imagen, and Lyria keep their dedicated APIs.
  • feat
    Google API mode switch — legacy generateContent as a safety escape hatch
    The Interactions API has no per-category safety thresholds — filtering is Google-managed. Settings → Backend & Safety → Google API now offers Interactions (current) and generateContent (legacy): legacy mode re-arms the threshold knobs for edgy artistic work and doubles as a wholesale behavioral rollback. The panel greys the knobs out (with an explanation) while Interactions is active; /api/health reports the mode; hosted instances pin it via SYNTH_GOOGLE_API.
  • fix
    Thinking-model output can no longer leak into streamed text
    Interactions streams thinking summaries as separate events; the new stream handlers (backend and ChatRoom) only surface text from real model output, and ChatRoom's truncation auto-continue now keys on status: incomplete. Verified live across text, streaming, JSON mode, chat history, vision, image gen, and web-search grounding in both API modes; new unit suite (tests/test_google_api.py) locks the store=false contract.

Backend Tiers, Safety Controls & Feedback
  • feat
    Local-model backend tier — run text generation on your own hardware, unrestricted

  • feat

    Safety blocks are now a typed error end-to-end (SafetyBlockedError → structured 422), so the error panel shows what was blocked and why, with a Report wrongly blocked button: prefilled GitHub issue form (wrongly_blocked.yml) and/or a local report to POST /api/feedback (JSONL on the operator's disk).   • security
    Phase-1 hardening: SSRF guard, upload consent, retention, rate limiting
    synth_fetch now validates URLs and every redirect hop against private/loopback/metadata addresses with a 10s timeout and 2 MB cap (workflow-engine/urlGuard.js); ANALYZE_URL gets a scheme/private-literal pre-check. First image upload shows a one-time consent notice (images go to Google's APIs — see Terms §7). Hosted mode (SYNTH_HOSTED=1): env-only API key, per-IP rate limiting (429 + Retry-After), and an hourly retention purge (default 30 days). None of this affects a normal local install.

Local-Disk Persistence — Auto-Backup & Restore
  • feat
    New /api/save-output generic endpoint — content-addressed JSON backup for any store
    POST {kind, content, filename_hint?}. Files land at Synthograsizer_Output/JSON/<Kind>/<slug>-<6-char sha256>.json. Server-side dedup is content-addressed (id + timestamps stripped before hashing) so repeated saves of the same logical content land on the same file. Supported kinds: agent_profile, taste_profile, agent_log, story_output, workflow_output (last three reserved for follow-ups). Sibling endpoints: GET /api/list-outputs/{kind} for picker UIs (returns lightweight per-kind summary), GET /api/get-output/{kind}/{filename} for full content, DELETE /api/delete-output/{kind}/{filename}.
  • feat
    Auto-backup wired into AgentProfileStore and TasteProfileStore
    Every .save() now fires a best-effort POST to /api/save-output alongside the local localStorage / IndexedDB write. Failures are silent (backend may be offline). Net effect: on first load of the Composer, every built-in agent gets re-saved through the store and back-filled to disk — instant historical backup. The user no longer loses agents to a wiped browser cache.
  • feat
    "Restore from Disk" modals on two surfaces
    Composer Library overflow menu (⋯) gets a "📁 Restore from disk…" item that lists every Agent Profiles/*.json with a one-click Restore button per row.
